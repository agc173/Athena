## 2026-04-09 (fase 2 i18n - hardening / rollback de runtime locale hack)
- Se elimina la implementación previa de `LocalAppLocale` que mutaba estado global de plataforma desde Compose (`Locale.setDefault + updateConfiguration` en Android y `AppleLanguages` en iOS).
- `AppLocaleEnvironment` se simplifica a un boundary de recomposición (`key(languageCode)`) sin side effects globales.
- Se mantiene intacta la base i18n ya válida: `AppLanguage`, persistencia local, `AppLanguageViewModel`, selector de idioma y bloque mínimo de strings traducidos por idioma.

Garantía actual:
- El estado vigente final usa recomposición controlada por `key(languageCode)` en `AppLocaleEnvironment`.
- No hay runtime locale forcing de plataforma (sin `Locale.setDefault`, sin `resources.updateConfiguration`, sin `AppleLanguages`).
- El cambio de idioma actualiza estado y recompone UI.
- **Limitación abierta:** todavía no está garantizado que `stringResource(...)` siga el idioma BWitch en todas las plataformas hasta definir una estrategia robusta/oficial para Compose MPP.

## 2026-04-09 (fase 2 i18n - consistencia temporal de navegación)
- Se audita la inconsistencia visual y se confirma una causa mixta: parte de la UI usaba `stringResource(...)` (atado al locale efectivo del sistema) y otra parte seguía en `Destination.title` hardcodeado en español.
- Se aplica estrategia conservadora temporal (Opción A) para navegación principal: títulos de top bar y labels de bottom tabs vuelven a usar una única fuente estable (`Destination.title`) mientras se define una solución robusta de runtime locale en Compose MPP.
- Se evita seguir extendiendo migración de textos en esta iteración; se mantiene infraestructura i18n (selector + persistencia + estado global) sin hacks nuevos de locale runtime.

Estado real final:
- La selección de idioma BWitch sigue persistiéndose y recomponiendo UI.
- La navegación principal ahora evita mezcla visual entre inglés/español por origen de strings distinto.
- `stringResource(...)` continúa dependiendo del locale efectivo del sistema hasta resolver estrategia oficial cross-platform.

## 2026-04-09 (horóscopo multiidioma end-to-end)
- Domain: `DailyHoroscope` incorpora `languageCode` y los contratos de horóscopo (`observe/get/pull`) pasan `languageCode` explícito para evitar defaults implícitos.
- Presentation: `HoroscopeViewModel` observa `AppLanguage` (fuente de verdad) y reinicia resolución de horóscopo por idioma al cambiar idioma o signo, manteniendo semántica de loading/error/refresh.
- Data local: cache Settings de horóscopo migra de `horoscope_daily_{date}_{sign}` a `horoscope_daily_{date}_{sign}_{lang}` con lectura legacy sin idioma como fallback explícito.
- Sync Firestore: `SyncHoroscopeDailyRepository` prioriza `horoscopeDaily/{date}/signs/{sign}/langs/{lang}` y solo cae al doc legacy `.../signs/{sign}` cuando no existe variante por idioma.
- Política de fallback explícita:
  - remoto legacy (`.../signs/{sign}`) se permite como fallback cross-language y se cachea bajo la clave del idioma solicitado para no romper `observe/get` por idioma;
  - local legacy (`horoscope_daily_{date}_{sign}`) solo se reutiliza para solicitudes `es` para evitar etiquetar en disco contenido histórico como otro idioma.
- Pull marker: persistencia del último pull se vuelve por idioma (`last_pulled_date_iso_{lang}`) para no bloquear refresh automático al cambiar lenguaje en el mismo día.
- Functions: `HoroscopeGenerator` escribe también `languageCode` en cada documento generado para facilitar trazabilidad del contenido.
- Docs Firestore: añadido schema de la ruta canónica multiidioma y fallback legacy.

Impacto arquitectónico:
- Sin cambios de módulos ni patrón base. Se completa propagación de idioma en la vertical de horóscopo (domain/data/presentation + backend) respetando Clean Architecture.

Pendientes:
